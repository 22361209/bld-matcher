# ADR 0007: Processing Module Decomposition

- Status: accepted
- Date: 2026-07-11
- Owners: BLD

## Context

询价 Excel、采购/销售合同 PDF 和材料持久化已经进入各自领域边界，但三个实现文件分别聚集了多种职责。继续在这些文件中叠加功能会扩大回归范围，使 Excel/PDF 输出兼容和材料数据安全越来越难以单独验证。

## Decision

1. 分三次独立提交拆分询价 Excel、合同 PDF 和材料持久化，每次只移动一个领域并先固定原行为。
2. 原有公共模块保留为兼容门面，旧导入路径和公共名称不变；新实现按读取、解析、规则、渲染、导入和 Repository 等职责进入所属领域子包。
3. 不改变页面 URL、表单字段、API、数据库 Schema、运行数据目录或生成文件命名。本次拆分不引入新框架和依赖。
4. Excel 以工作表、表头、行、匹配摘要和价格列为兼容合同；PDF 以页数、关键文字、金额和签章区为兼容合同；材料以 CRUD、搜索、统计、导入原子替换和失败恢复为兼容合同。
5. 完成拆分的聚集文件从遗留行数白名单删除。兼容门面硬限制 80 行，拆出的单个处理模块硬限制 360 行，并纳入 Pyright；后续新增职责必须继续拆分，不能回流门面。

## Alternatives

- 只冻结原文件行数：可以阻止继续增长，但不能降低修改冲突和测试范围。
- 同时重写行为与拆文件：难以判断 Excel、PDF 或材料数据差异来自移动还是业务变化。
- 引入新的文档或持久化框架：当前没有明确收益，且会扩大部署和兼容风险。

## Consequences

- 旧调用方无需一次迁移，兼容门面可以在消费者逐步切换后通过单独 ADR 删除。
- 文件数量增加，但职责、类型检查和测试入口更明确；每个阶段都可以独立回退。
- 后续修改输出格式或 Schema 必须另立变更，不得混入结构拆分。

## Verification

- 每阶段运行对应领域回归、兼容门面测试、Ruff、Pyright 和项目合同检查。
- Excel 验证 `.xls` 与 `.xlsx` 的工作表和追加列；合同验证采购与销售 PDF 的页数和关键内容；材料验证数据库 Schema 不变、文件导入和失败恢复。
- 三阶段完成后运行 `uv run python scripts/verify.py --base-ref main`，重启本机服务并执行页面与健康探针验收。
