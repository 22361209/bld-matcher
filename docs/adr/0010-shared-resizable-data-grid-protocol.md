# ADR 0010: Shared Resizable Data Grid Protocol

- Status: accepted
- Date: 2026-07-18
- Owners: BLD

## Context

产品目录、材料明细、管件资料和报价记录都需要展示宽表格。此前各页面分别处理横向滚动、固定表头、分页和个别列宽调节，导致页面纵向空间利用率不一致，列宽能力不完整，分页与表格视觉上彼此分离。页面之间继续复制实现会增加交互冲突和维护成本。

这些列表仍由服务端筛选和分页；共享表格框体只能负责展示与浏览器交互，不能改变领域 Service、Repository 或 URL 查询合同。

## Decision

1. 多列主数据列表统一使用 `data-resizable-grid` 框体、`data-grid-scroll` 滚动区和 `_data_grid_footer.html` 底栏。共享样式与脚本放在 `static/components/` 并由 `base.html` 加载。
2. 表头固定在框内滚动区顶部，表体支持上下和左右滚动，表头与表体使用浅色 1px 竖向分隔线。底栏固定在滚动区之外，展示当前记录范围、筛选总数和服务端分页。
3. 每个实际列对应一个稳定的 `data-col` 键。桌面端在列头右缘提供独立调宽命中区，避免与筛选、排序和列拖拽共享事件目标；同时支持方向键调整、双击单列重置和底栏整体重置。
4. 列宽只保存在当前浏览器的 `localStorage`，存储键同时包含页面标识、版本和登录用户作用域。列宽不进入数据库、URL、API 或审计日志。
5. 指针抬起、取消、捕获丢失和窗口失焦都必须经过同一幂等清理流程，恢复页面选择和鼠标状态后再持久化最终宽度。
6. 服务端分页仍负责校正越界页码；页面展示的当前范围必须与实际返回记录来自同一个已校正页。
7. 本次验收以桌面端鼠标和键盘交互为范围；窄屏继续保留框内横向滚动和现有响应式底栏，触屏调宽体验留待后续单独决策。

## Alternatives

- 在四个页面分别实现：会继续复制滚动、分页和列宽状态逻辑，也更容易让筛选、列拖拽与调宽事件互相干扰。
- 引入第三方数据网格：当前需求不需要客户端数据模型、虚拟滚动或新的构建链，新增依赖和迁移成本超过收益。
- 继续依赖整个页面滚动和页面外分页：表头会离开视野，底部控制占用额外纵向空间，无法达到本次空间利用目标。

## Consequences

- 四个列表获得一致的固定表头、双向滚动、底栏统计、列宽调节和视觉分隔线。
- 页面模板需要提供完整 `colgroup`、稳定列键和分页上下文；列数不一致时共享脚本拒绝初始化，以避免错误写宽度。
- 浏览器本地存储不可用时，当前页面仍可调整列宽，但刷新后不会保留。
- 新增共享组件代码和真实浏览器验收责任；后续新增主数据宽表格时必须复用这一协议或通过新的 ADR 修改它。

## Verification

- 项目合同检查确认四个主列表声明共享框体、滚动区和底栏，并且共享资产只从 `base.html` 加载。
- JavaScript 回归覆盖宽度边界、存储数据归一化，以及抬起、取消、捕获丢失和窗口失焦的幂等清理。
- Web 回归覆盖管件越界页码重新加载真实末页，底栏范围与实际记录一致。
- 真实桌面浏览器验收四页的固定表头、双向滚动、浅色竖线、列宽调整、持久化、重置和控制台无错误。
