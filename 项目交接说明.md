## 系统用途

这是一个 Flask 网页系统，主要用于：

1. 客户询价处理
2. 产品目录维护
3. 产品目录导入 / 导出
4. 产品图片和含税单价维护
5. 操作日志查看
6. 账号和权限管理
7. 生产料单生成
8. 冲压材料明细维护

## 当前主要页面

### 询价处理

入口：`/`

功能：

- 页面主操作为上传客户询价 Excel 并生成匹配结果，支持点击选择文件或把 Excel 直接拖入主输入区域
- 支持 `.xls` 和 `.xlsx`
- 支持无表头文件，上传后可选择匹配列
- 支持一个单元格中多个 OE 号，用逗号、斜杠、分号、换行等拆分
- 不按普通空格拆分，因为部分 OE 号本身含空格
- 匹配后导出 Excel，保持客户源文件格式
- 输出文件新增 `BLD NO.` 和 `匹配说明` 列
- 文件名格式为 `re + 当前日期 + 源文件名`
- 历史报价文件默认折叠隐藏，点击“历史报价文件”后展开
- 历史报价文件展开后可按文件名、日期、关键词或操作用户搜索
- 历史报价文件列表显示文件名、格式、操作用户、生成时间和下载入口
- 支持直接在首页“客户询价源文件 / OE、品牌号码或 BLD 号”输入框里录入号码快速查询；输入 4 位纯数字时会按 BLD 号片段返回多个可能型号
- 未选择文件时，该输入框用于 OE、品牌号码、完整 BLD 号或 4 位 BLD 数字查询；点击选择或拖入文件后显示文件名并用于上传匹配
- 未选择文件且粘贴多个号码时，会自动生成临时询价表并进入匹配结果页，可下载 Excel 和图纸包；空格、逗号、分号、斜杠、顿号、换行均可作为多个号码分隔
- 询价匹配可选择是否在导出的 Excel 中追加单价列；支持追加产品目录中的含税单价、不含税单价，或填写汇率后追加美金价
- 不含税单价计算规则：`含税单价 ÷ 1.1` 后四舍五入到整数；美金价计算规则：`含税单价 ÷ 1.1 ÷ 汇率`，结果保留两位小数
- 快速号码查询结果里的产品图片在当前页面浮层预览，不打开新页面
- 选择匹配列后先查看网页结果，点击“下载 Excel”时才生成 Excel 文件
- 选择匹配列页和结果页提供“返回上一步”，不再显示无意义的“返回首页”

### OpenClaw 内部 API

文档：`OPENCLAW_API.md`

用途：

- 给 OpenClaw 里的机器人调用，不绑定具体机器人名称
- 管理员在右上角菜单的“内部 API Key”页面生成 Key；完整 Key 只显示一次，可重新生成或停用
- `/api/internal/*` 必须带 `Authorization: Bearer <key>`，不允许匿名调用
- 支持截图/OCR 后的号码数组、纯文字号码、客户原始 Excel 文件
- 机器人只需要调用 API，拿结构化摘要和本机结果文件绝对路径
- 不替代现有网页，网页继续给人工使用

接口：

- `POST /api/internal/inquiry/numbers`：号码数组或文字号码查询，可显式导出新的结果 Excel
- `POST /api/internal/inquiry/file`：本机文件路径或上传 Excel，可显式在客户源文件基础上追加结果列
- `POST /api/internal/inquiry/analyze`：仅分析命中情况，不导出文件

默认行为：

- `numbers` 和 `file` 不传 `export` 时默认只分析，不生成文件
- 需要输出 Excel 时必须显式传 `export: true`
- `file_path` 只允许读取项目目录、`uploads/`、`outputs/` 里的 `.xls/.xlsx` 文件，范围外路径会拒绝
- 导出默认写入 `outputs/openclaw/`
- 导出文件名统一为：`reYYMMDD_源文件名称.xls/xlsx`
- 客户原始文件增强使用上传文件或 `file_path` 的源文件名；号码数组/文字号码没有源文件，导出前必须由机器人询问并传 `source_name`
- 系统会去掉路径、强制正确后缀，重名自动追加 `_2`、`_3`

返回：

- `ok`、`mode`、`summary`、`matched_count`、`unmatched_count`
- `rows`、`unmatched_list`、`output_path`、`output_name`
- `output_path` 为本机绝对路径

价格：

- `price_mode=tax`：含税单价
- `price_mode=net`：不含税单价，按 `含税单价 ÷ 1.1` 后四舍五入到整数
- `price_mode=usd`：美金价，需要 `exchange_rate`
- `price_mode=none`：不导出价格列

### 生产料单

入口：`/materials`

功能：

- 上传生产计划 Excel，生成冲压生产料单
- 下载生产计划模板
- 查看材料明细列表
- 按母件编码 / 零件编码 / 规格尺寸搜索
- 启用 / 停用状态筛选
- 新增材料明细，当前为生产料单页内浮层入口
- 编辑材料明细
- 停用材料明细
- 导入新的冲压料单 Excel，重建材料明细库

生成料单时现在读取数据库中的启用材料明细，而不是直接读取 Excel 文件。网页上编辑、新增、停用材料明细后，会影响下一次生成结果。

材料字段说明：

- 母件编码：用于匹配生产计划原来的“型号”
- 零件编码：如 `KA8234`，网页新增/编辑时必填
- 类别
- 车型
- 零件名称：网页新增/编辑时必填
- 下料只数：网页新增/编辑时必填
- 规格尺寸：网页新增/编辑时必填，统一显示为 `厚度×宽度×长度`
- 单件重量kg：列表中按 `宽度 × 长度 × 7.85 × 厚度 ÷ 下料只数 ÷ 1,000,000` 显示，两位小数

网页新增/编辑材料明细时，规格尺寸只填写一个输入框，支持：

- `2.5 357 1260`
- `2.5*357*1260`
- `2.5-357-1260`
- `2.5/357/1260`

保存后会自动拆成厚度、宽度、长度参与后续料单计算，并统一显示为 `2.5×357×1260`。

材料明细搜索规则：

- 输入 1 个规格值：匹配厚度、宽度、长度任意一个
- 输入 2 个规格值：按规格顺序匹配，允许中间缺省，例如 `357/1260` 匹配宽度和长度
- 输入 3 个规格值：完整匹配厚度、宽度、长度
- 分隔符支持空格、`/`、`-`、`*`、`×`
- 搜索提交后会定位到 `#materials-results`，避免回到页面顶部

材料明细录入效率约定：

- 必填项排在前面：母件编码、零件编码、零件名称、下料只数、规格尺寸
- 非必填项排在后面：类别、车型
- 回车键在材料表单中跳到下一个输入框
- `Shift + Tab` 回到上一个输入框
- `Command + Enter` 和 `Ctrl + Enter` 保存表单

生产料单输出包含：

- `总汇总`
- `料单明细`
- `按规格汇总`
- `生产需求`
- 如有未匹配型号，额外生成 `检查提醒`

### 合同管理

入口：`/contracts`

功能：

- 提供采购合同和销售合同两个入口；销售合同已可生成 PDF
- 手动填写甲方、乙方、双方联系人、双方电话、交货地点、付款方式、质量要求和价格说明
- 甲方默认带入“玉环博莱德机械有限公司”
- PDF 抬头甲乙双方均显示公司名称、联系人和电话；交货地点只放在交货条款中
- 页面按 A4 合同纸面组织，合同抬头、正文条款和签章区与 PDF 基本一致；明细表为了录入保留更宽的表格区域
- 页面可录入位置使用独立颜色提示；签章区不显示统一社会信用代码；甲乙双方的地址、电话、开户行、账号和签章日期均为可选填写，留空也可生成 PDF
- 在明细表中填写 BLD号、数量、单价、交期和备注，页面实时预览金额和合计；OE号、产品名称和适用车型会按 BLD号从产品目录自动带入；销售合同明细额外包含客户编码，且单价为空时会带入目录含税单价
- 明细行支持在页面里新增和删除
- 采购合同 PDF 明细表包含 BLD号、OE号、产品名称、适用车型、数量、单价、金额、备注和交期；销售合同 PDF 明细表额外包含客户编码

当前合同管理是轻量生成工具，不维护供应商/客户库、审批状态或合同台账。

### 多用户文件隔离

上传和输出文件现在按用户目录隔离：

```text
uploads/u用户ID-用户名/
outputs/u用户ID-用户名/
```

生成文件名会带用户名。例如：

```text
outputs/u1-007/re260429-007-客户询价.xlsx
outputs/u1-007/catalog-export-bld-007-260429.xlsx
```

普通用户只显示和下载自己的输出文件。管理员在最近结果和最近料单里可以看到所有用户目录和旧根目录输出，并可通过下载接口访问 `outputs/` 下的文件。

会修改全局数据的操作已加导入锁：

- 产品目录导入 `/catalog`
- 材料数据导入 `/materials/data`
- 单价确认导入 `/prices/import/apply`

当一个用户正在执行上述导入时，其它用户再执行同类全局导入会收到“当前已有用户正在执行导入操作，请稍后再试”提示。

### NAS 更新唯一流程

不要再手动上传 zip 覆盖根目录。NAS 更新必须通过 Git 部署：

```text
本机代码 -> git push nas main -> NAS 运行目录 git pull/reset -> docker-compose 重建
```

禁止使用 Finder、File Station、`scp`、`rsync`、压缩包或手动拖拽直接覆盖 `/volume1/docker/bld-matcher` 的程序文件。直接覆盖会让运行目录脱离 Git 管理，也容易误碰运行数据。

如果本机 Git 仓库异常、无法正常 push，应先修复本机仓库，或从 NAS Git 仓库干净克隆到临时目录后提交推送；仍然不要直接覆盖 NAS 运行目录。

本机开发完成并 commit 后：

```bash
cd "/Users/linzhenyue/Projects/bld-matcher"
git push nas main
```

NAS 上更新代码并重启容器：

```bash
ssh -i ~/.ssh/bld_matcher_deploy deploy@192.168.110.93
cd /volume1/docker/bld-matcher
git fetch origin main
git reset --hard origin/main
git status -sb
sudo /usr/local/bin/docker-compose up -d --build
sudo /usr/local/bin/docker-compose exec -T bld-matcher python tools/generate_product_thumbnails.py
sudo /usr/local/bin/docker-compose ps
```

如果出现容器名冲突：

```bash
sudo /usr/local/bin/docker rm -f bld-matcher
sudo /usr/local/bin/docker-compose up -d --build
```

注意：

- `data/products.sqlite3`、`data/catalog.xlsx`、`data/stamping_materials.xlsx` 不由 Git 管理，`git pull` 不会覆盖它们。
- `uploads/` 和 `outputs/` 不由 Git 管理。
- `.env` 是 NAS 本机运行配置，不进入 Git，不要用本机 `.env` 覆盖。
- NAS 运行目录更新后，`git status -sb` 应只显示 `## main`；如果出现大量 `M` 或 `??`，先查明原因，不要继续重建。
- Docker 重启目前需要输入 `deploy` 的 sudo 密码。
- 需要输入 sudo 密码时，必须打开可见 macOS Terminal 窗口让用户输入，不要在隐藏执行会话里运行会卡住的 sudo 命令。
- NAS 当前使用 Gunicorn + Docker Compose 运行，`restart: unless-stopped` 已生效；这和本机临时 Flask 开发进程不同，容器异常退出时会自动重启。
- 产品列表缩略图在 `data/product_images/thumbs/` 下按 NAS 运行数据生成，不进入 Git；产品图片批量更新或部署后可运行 `python tools/generate_product_thumbnails.py` 预生成。
- NAS 防火墙限制 SSH 只允许内网访问这件事还没配置，计划后续处理。

如果部署后还是老版本，重点检查：

- 当前打开的网址是否是 NAS 地址，而不是本机 `127.0.0.1`
- NAS 运行目录是否已经 `git pull` 到最新 commit
- Container Manager/Docker 是否仍然运行旧容器或旧映像
- 浏览器是否缓存，可强制刷新

## 给新接手 Codex 的提示

如果在新电脑或新对话中继续开发，请先阅读：

```text
AGENTS.md
PROJECT_BRIEF.md
```

然后检查：

```bash
pwd
rg --files
sed -n '1,140p' app.py
sed -n '1,220p' app/database.py
sed -n '1,220p' PROJECT_BRIEF.md
git remote -v
```

本文件是历史档案，不建议默认整篇读取。需要查旧决策时用 `rg` 搜索关键词，再读取相关小节。

不要第一时间重构。这个项目是用户持续迭代出来的业务工具，很多设计来自实际使用习惯，优先保留已有行为。

项目约定：

- 每次重要改动优先更新 `PROJECT_BRIEF.md` 中的当前状态。
- 需要在网页“系统更新”展示的变更，再同步更新 `项目交接说明.md` 的“当前最近重要变更”。
- “当前最近重要变更”使用 `### YYYY-MM-DD · commit · 标题` 格式；一次 commit 对应一条版本更新，同一天多次更新就写多条。
- 重要改动包括新增/删除功能页面、数据库结构或迁移、部署流程、权限逻辑、数据文件归属、Excel 导入导出格式、核心业务规则和外部依赖变化。
- 完成重要改动后要把代码和文档一起 commit。

修改前要注意：

- 不要覆盖 `data/products.sqlite3`
- 不要覆盖 `data/catalog.xlsx` 和 `data/stamping_materials.xlsx`
- 不要删除 NAS 上已有数据
- 页面更新后要重启 Flask 服务
- NAS 更新时必须先 `git push nas main`，再在 NAS 运行目录 `git fetch origin main && git reset --hard origin/main` 并重建 Docker 映像
- 不要用 `scp`、`rsync`、File Station 或压缩包直接覆盖 NAS 运行目录的程序文件，除非用户明确要求应急恢复，并且之后必须立即把 Git 状态恢复干净
- 当前主要页面已统一到低对比工作台风格，视觉调整尽量小步迭代，优先贴合用户实际使用习惯
